# Slave Artifact Handoff

슬레이브 작업의 완료는 결과를 생성하거나 대화로 설명한 시점이 아니다. 마스터가
직접 검증할 수 있도록 `output.md`와 실제 작업 결과물을 공유 작업공간에 저장하고,
그 위치를 마스터에게 전달해야 완료된다.

이 문서는 마스터가 슬레이브 명령용 프롬프트를 작성할 때 포함해야 하는 공통 인계
계약이다. [@agent-instructions]

## 기본 구조

마스터는 기존 방식대로 `input.md`를 전달한다. 슬레이브는 그 파일이 있는 작업
디렉터리에 다음 구조로 결과를 작성한다.

```text
<task-directory>/
├─ input.md
├─ output.md
└─ artifacts/
   ├─ <작업 결과물>
   └─ ...
```

별도 지시가 없다면 `input.md`가 위치한 디렉터리를 `<task-directory>`로 사용한다.

## 슬레이브 작업 절차

1. `input.md`를 완전히 읽고 요구사항, 제한사항과 완료 조건을 확인한다.
2. 필요한 작업을 수행한다.
3. 생성하거나 수정한 최종 결과물을 `<task-directory>/artifacts/`에 저장한다.
4. 각 결과물이 실제로 존재하고 크기가 0보다 큰지 확인한다.
5. 각 결과물의 파일 크기와 SHA-256을 계산한다.
6. 수행 내용과 결과물 목록을 `output.md`에 기록한다.
7. `output.md`와 모든 결과물이 준비된 뒤에만 완료를 보고한다.

`output.md`는 인계 준비가 끝났음을 나타내는 마지막 파일로 작성한다. 결과물 생성이
진행 중인 상태에서 먼저 작성하거나 완료를 보고하지 않는다.

## 필수 인계 규칙

- 결과물을 대화, 도구 출력 또는 임시 위치에만 남기지 않는다.
- 마스터에게 전달할 파일은 반드시 `artifacts/` 아래에 저장한다.
- `output.md`에는 모든 결과물의 상대경로, 용도, 크기와 SHA-256을 기록한다.
- 존재하지 않는 파일이나 임시 파일을 결과물로 보고하지 않는다.
- 생성 도구가 응답으로만 결과를 반환하면 공유 작업공간의 파일로 영속화한다.
- 결과물을 파일로 영속화할 수 없다면 완료로 보고하지 않고 `BLOCKED`로 보고한다.
- 마스터가 결과물을 받지 못하면 재생성하지 않는다. 먼저 기존 결과물을 찾아 같은
  파일을 다시 인계한다.
- 재생성이 필요할 때는 마스터의 명시적인 지시를 기다린다.
- 원래 `input.md`와 기존 결과물을 삭제하거나 덮어쓰지 않는다.
- 마스터가 파일명이나 저장 위치를 지정했다면 그 지시를 우선한다.

## output.md 계약

```markdown
# 작업 결과

## 작업 정보

- 작업 ID: `<task-id>`
- 상태: `COMPLETED` 또는 `BLOCKED`
- 입력 파일: `input.md`

## 수행 내용

- 수행한 작업
- 중요한 판단 또는 변경 사항

## 결과물

| 경로 | 설명 | 크기 | SHA-256 |
|---|---|---:|---|
| `artifacts/<filename>` | 결과물 설명 | `<bytes>` | `<sha256>` |

## 요구사항 확인

| 요구사항 | 결과 | 근거 |
|---|---|---|
| `<input.md 요구사항>` | `PASS`, `FAIL`, `NOT_VERIFIED` | `<파일 또는 확인 내용>` |

## 검증 내용

- 수행한 검증 명령과 결과
- 파일 존재 여부
- 결과물 형식, 크기 또는 해상도 등의 기본 검사

## 미완료 사항 및 위험

- 없으면 `없음`
- 있으면 원인과 마스터가 결정해야 할 내용을 기록
```

## 완료 보고 계약

슬레이브는 파일 인계 준비가 끝난 경우에만 다음 형식으로 마스터에게 보고한다.

```text
TASK_OUTPUT_READY
task_id: <task-id>
status: COMPLETED
output_md: <output.md 절대경로>
artifacts_dir: <artifacts 디렉터리 절대경로>
artifact_count: <결과물 개수>
```

완료하지 못했거나 결과물을 저장할 수 없다면 다음 형식을 사용한다.

```text
TASK_OUTPUT_BLOCKED
task_id: <task-id>
status: BLOCKED
output_md: <작성했다면 output.md 절대경로>
reason: <완료 또는 저장이 불가능한 구체적인 이유>
```

`TASK_OUTPUT_READY`는 `output.md`와 모든 결과물이 실제로 존재하며 마스터가 접근할
수 있을 때만 사용할 수 있다.

## 마스터 수신 규칙

마스터는 완료 보고를 받으면 `output.md`를 읽고 각 결과물의 존재 여부, 크기와
SHA-256을 직접 확인한 뒤 결과물을 검증한다. 전달된 파일이 보이지 않거나 해시가
다르면 결과물을 직접 재생성하지 않고 먼저 기존 슬레이브에게 재인계를 요청한다.

마스터가 동일한 파일을 확인하기 전까지 슬레이브 작업은 수신 완료로 처리하지 않는다.

## 서로 다른 PC에서의 인계

슬레이브와 마스터가 공유 파일시스템을 사용하지 않으면 이 문서에 기록된 슬레이브 PC의
절대경로만으로는 인계가 완료되지 않는다. `output.md`와 `artifacts/`를 외부 전달용
ZIP으로 묶고 ZIP 크기와 SHA-256, manifest 및 마스터 실행 프롬프트를 함께 전달한다.

패키징, 사용자 전송, 마스터 staging 검사와 재인계 절차는
[Cross-PC Slave Handoff](cross-pc-slave-handoff)를 따른다. ZIP을 마스터 PC로 실제
복사하고 마스터가 동일 바이트를 검증하기 전까지 `TASK_OUTPUT_READY`는 슬레이브 측
준비 완료를 뜻할 뿐 마스터 수신 완료를 뜻하지 않는다.

같은 신뢰 가능한 사설 Wi-Fi/LAN에서는 일회용 token으로 제한된 수신기와 송신기를
사용해 네 전달 파일을 무선으로 옮길 수 있다. 무선 전송 성공은 파일 수신 완료일 뿐
결과 검증이나 단계 완료 판정을 대신하지 않는다.
