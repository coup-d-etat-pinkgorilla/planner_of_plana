# Frontend Section Direction and User Flows

이 문서는 두 종류의 결정을 분리해 기록한다.

1. **확정된 시각·동작 규칙**: 모든 기본 탭의 실제 섹션 구현이 따라야 하는 계약이다.
2. **검수 전 사용자 흐름 가설**: 계획 탭을 기준으로 나머지 탭의 기능을 사용자 목적별로
   나눈 것이다. 화면의 열 수, 정확한 크기와 배치는 사용자가 흐름을 검수한 뒤 정한다.

여기서 **기믹**은 장식 효과가 아니라 사용자가 하나의 목적을 달성하는 기능 묶음을
뜻한다. 예를 들어 학생 탭의 `학생 탐색`과 `계획에 추가`는 서로 이어질 수 있지만 각각
독립적으로 이해되고 완료될 수 있는 기믹이다.

현재 기본 탭은 홈, 학생, 계획, 인벤토리, 전술대항전, 통계, 스캔, 설정이며
Adaptive-Sync 진단은 설정에서 진입하는 보조 화면이다. [@app-sections]

## 1. 확정된 프론트엔드 섹션 규칙

### 1.1 탭과 섹션의 기본 구조

- 모든 탭은 위쪽에 해당 탭의 헤더를 가진다.
- 헤더 아래에는 탭의 기능에 따라 하나 이상의 섹션을 둔다.
- 기능을 한 섹션에 밀어 넣지 않는다. 사용자의 목적과 행동 전환점이 달라지면 별도
  섹션 후보로 나눈다.
- 섹션은 헤더 아래의 전체 높이를 반드시 혼자 차지하지 않는다. 같은 쪽 공간을 위·아래
  섹션이 나누어 쓸 수 있다.
- 중앙 영역도 위·아래 평행사변형 등으로 분할할 수 있다. 다만 어느 경계에도 연결되지
  않은 채 중앙에 독립적으로 떠 있는 섹션은 만들지 않는다.

### 1.2 80도 사선 형상

- 완전한 정사각형 또는 직사각형 섹션은 만들지 않는다. 모든 섹션은 적어도 한 면에
  80도 사선 처리를 가진다.
- 왼쪽 경계에 붙는 섹션은 오른쪽 면에 사선을 둔다.
- 오른쪽 경계에 붙는 섹션은 왼쪽 면에 사선을 둔다.
- 중앙에서 양옆 섹션 사이를 잇는 섹션은 양쪽 면에 사선을 둔다.
- 좌우 가장자리에 좁은 잔여 공간이 생기고 독립 기능 또는 상태 표시에 적합하면 삼각형
  섹션을 사용할 수 있다.
- 모든 섹션은 둥근 모서리와 투명 글라스 표면을 가진다. 사선 클리핑 이후에도 둥근
  모서리, 테두리와 그림자가 실제 실루엣을 따라야 한다.

80도는 홈 메뉴에서 이미 사용한 사선 기하와 같은 기준이며, 내부 콘텐츠는 사선이
차지하는 폭을 제외한 안전 영역 안에 있어야 한다. [@diagonal-home-migration]

### 1.3 내부 요소 배치

- 먼저 섹션 안의 기능을 자연스러운 **행 단위**로 나누고, 각 행의 높이와 순서를 정한다.
- 같은 행의 컨트롤은 그 Y 위치에서 실제로 남는 가용 폭을 기준으로 간격과 크기를
  조절한다. 평행사변형이나 사다리꼴의 직사각형 bounding box만 기준으로 삼지 않는다.
- 사선의 좁아지는 방향에서 라벨, 버튼, 입력기가 잘리거나 과도하게 압축되면 행을
  분리하거나 기능을 다른 섹션으로 옮긴다.
- 한 섹션에 들어가는 요소 수보다 사용자가 한 번에 이해해야 하는 결정 수를 줄이는 것을
  우선한다.

### 1.4 부착면과 전환 방향

각 섹션은 화면에서 붙어 있는 면을 기준으로 등장하고 퇴장한다. 각도는 모니터 공간의
수학 방향을 사용하며 `0`은 오른쪽, `180`은 왼쪽이다. [@transition-migration]

| 섹션의 부착면 | intro | outro |
|---|---:|---:|
| 왼쪽 | 0 | 180 |
| 오른쪽 | 180 | 0 |
| 아래 | 80 | 260 |
| 위 | 260 | 80 |

- 위·아래로 나뉜 섹션은 각자의 실제 부착면을 사용한다.
- 중앙 섹션을 띄우지 않는 이유는 시각적 정렬뿐 아니라 어느 면에서 들어오고 나가는지
  모호해지는 것을 막기 위해서다.
- 여러 면에 닿는 섹션은 실제 등장 근거가 되는 주 부착면을 디자인 시 명시한다.
- 현재 placeholder의 임시 모션 값은 최종 계약으로 보지 않는다. 실제 탭 섹션을 설계할
  때 이 표를 적용한다.

## 2. 사용자 흐름을 섹션으로 바꾸는 기준

아래 흐름은 섹션의 위치나 크기를 말하지 않는다. 실제 디자인 시에는 각 기능 그룹마다
다음 질문에 답한 뒤 섹션을 만든다.

1. 사용자가 이 기능에 들어올 때 이미 알고 있거나 선택한 대상은 무엇인가?
2. 이 기능에서 사용자가 내려야 하는 핵심 결정은 하나인가?
3. 완료를 판단할 수 있는 결과 또는 저장 상태는 무엇인가?
4. 다음 행동은 같은 탭 안에 있는가, 다른 탭으로 넘어가는가?
5. loading, empty, error, disconnected 또는 review-required 상태가 정상 흐름을 어떻게
   바꾸는가?

목록 탐색, 상세 확인, 편집, 검토, 결과 분석은 서로 데이터가 이어져도 사용자의 사고
단계가 다르면 별도 기믹으로 취급한다. 반대로 검색·정렬·필터처럼 같은 대상을 찾기 위한
보조 행동은 하나의 탐색 기믹 안에 묶는다.

## 3. 계획 탭 — 사용자가 정한 기준 사례

계획 탭의 상세 구성은 사용자가 별도로 기획한다. 이 문서에서는 다른 탭을 나누는 기준이
되는 큰 사용처만 기록한다.

- **계획 통계**: 현재 시나리오, 저장된 계획, 미계획 학생 분석 등 계획 관련 통계를 본다.
- **그룹별 빠른 계산**: 사용자가 정한 그룹을 범위로 선택해 필요량을 빠르게 계산한다.
- **프리셋 설정**: 학생을 계획에 넣을 때 적용할 목표 프리셋을 만든다.
- **계획 편집·결과**: 다중 학생의 현재와 목표를 비교하고 학생별·전체 총 필요량,
  인벤토리 기반 부족량을 확인하며 계획을 저장·복원한다. [@p6-workflow]

나머지 탭은 계획 탭의 기능을 복제하지 않고, 필요한 대상이나 결과를 계획 탭으로
넘겨주는 방향으로 흐름을 잡는다.

## 4. 홈 탭

### 기능 그룹

- **현재 작업 맥락 확인**: 선택 프로필, backend와 게임 창 연결, 마지막 스캔 시각을 본다.
- **주의가 필요한 상태 확인**: 검토 대기 스캔, 연결 오류, 저장 실패, 오래된 데이터처럼
  다음 행동을 막거나 신뢰도를 낮추는 상태를 찾는다.
- **요약 확인**: 학생·인벤토리 수, 활성 계획, 주요 부족 재화와 최근 스캔 결과를 훑는다.
- **빠른 재개**: 학생 탐색, 계획, 인벤토리 분석, 스캔 등 목적 탭으로 바로 이동한다.

### 예상 주 흐름

1. 사용자가 앱을 열거나 홈으로 돌아온다.
2. 현재 프로필과 연결·데이터 최신 상태를 확인한다.
3. 차단 또는 검토 대기 상태가 있으면 해당 설정이나 스캔 흐름으로 이동한다.
4. 문제가 없으면 활성 계획과 부족 요약에서 이어서 할 일을 고른다.
5. 선택한 목적 탭으로 이동하고, 홈은 상태 요약과 재개 지점 역할을 끝낸다.

### 분기와 탭 간 인계

- 프로필 또는 backend 문제 → 설정
- 데이터가 없거나 오래됨 → 스캔
- 검토 대기 후보가 있음 → 스캔의 결과 검토
- 계획 진행 상황을 바꿈 → 계획
- 부족 재화 원인을 자세히 봄 → 인벤토리
- 특정 학생부터 확인함 → 학생

### 사용자 검수 포인트

- 첫 시선의 우선순위가 `문제/검토 상태`인지 `계획 진행 요약`인지 확인해야 한다.
- 홈을 일상적인 대시보드로 쓸지, 작업을 시작하는 빠른 런처로 쓸지 비중을 정해야 한다.

## 5. 학생 탭

### 기능 그룹

- **학생 탐색**: 학생 목록을 보고 이름·ID·태그 검색, 필터와 정렬로 대상을 좁힌다.
- **학생 현재 상태 확인·수정**: 선택 학생의 정적 정보와 현재 육성 상태를 구분해 보고,
  허용되는 현재값을 수정한다.
- **스캔 신뢰도 검토**: 학생에 연결된 낮은 confidence 후보와 근거를 보고 수정·확정하거나
  보류한다.
- **계획으로 보내기**: 선택 학생에게 계획 프리셋을 골라 적용하고 계획에 추가한다.

### 예상 주 흐름 A — 목록과 상세 조회

1. 사용자가 검색·필터·정렬로 학생 범위를 정한다.
2. 목록에서 학생을 선택한다.
3. 정적 메타데이터와 스캔된 현재 육성 상태를 확인한다.
4. 다른 학생을 계속 비교하거나 필요한 현재값을 수정하고 저장한다.

### 예상 주 흐름 B — 계획에 추가

1. 목록 또는 상세에서 학생을 선택한다.
2. `계획에 추가`를 선택한다.
3. 적용할 계획 프리셋을 고른다. 기본 프리셋이 있으면 명시적으로 표시한다.
4. 추가될 목표를 확인하고 확정한다.
5. 같은 학생 탭에서 계속 탐색하거나 계획 탭으로 이동해 세부 목표와 계산 결과를 본다.

### 예상 보조 흐름 — 스캔 후보 검토

1. 검토가 필요한 학생 표시를 선택한다.
2. 기존 확정값, 새 후보값, confidence와 인식 근거를 비교한다.
3. 필드를 수정·확정하거나 후보를 보류한다.
4. 확정된 현재값으로 상세와 관련 계산이 갱신된다.

낮은 confidence 결과는 검토 없이 현재 상태가 될 수 없으며, 정적 메타데이터와 현재값,
계획 목표를 같은 필드로 합치지 않는다. [@migration-baseline]

### 사용자 검수 포인트

- 학생 선택 뒤 기본 초점이 `상세 조회`인지 `계획 추가`인지 정해야 한다.
- 프리셋 선택을 계획 추가 전 필수 단계로 둘지 기본값으로 빠르게 통과시킬지 정해야 한다.
- 학생 탭의 현재값 수정과 스캔 탭의 후보 수정 사이 책임 경계를 확인해야 한다.

## 6. 인벤토리 탭

### 기능 그룹

- **보유량 탐색**: 장비·아이템 모드와 카테고리, 검색, 정렬, zero-fill 규칙으로 재화를 찾는다.
- **보유량 조회·수정**: 선택 재화의 현재 보유량과 마지막 갱신 근거를 확인하고 수동 수정한다.
- **스캔 결과 비교·확정**: 기존 보유량과 새 스캔 후보를 비교해 수정·확정한다.
- **계획 부족 분석**: 계획 필요량, 보유량, 부족량과 영향을 받는 학생을 확인한다.

### 예상 주 흐름 A — 보유량 확인·수정

1. 장비 또는 아이템 범위를 고른다.
2. 카테고리·검색·정렬로 재화를 찾는다.
3. 재화를 선택해 현재 보유량, 데이터 유무와 갱신 시점을 본다.
4. 필요하면 수량을 수정하고 저장한다.
5. 변경으로 영향을 받는 부족량을 확인한다.

### 예상 주 흐름 B — 부족 원인 분석

1. 계획 부족 요약에서 부족 재화를 고르거나 목록에서 재화를 선택한다.
2. `부족 여부 → 부족량 → 총 필요량과 보유량 → 영향 학생` 순으로 근거를 확인한다.
3. 영향 학생과 계획 우선순위를 검토한다.
4. 목표를 바꾸려면 계획 탭으로, 현재 수량을 새로 얻으려면 스캔 탭으로 이동한다.

### 예상 주 흐름 C — 스캔 반영

1. 새 인벤토리 스캔 결과가 있음을 확인한다.
2. 바뀐 항목, 낮은 confidence 항목과 누락 항목을 구분해 비교한다.
3. 필요한 수량을 수정하고 확정한다.
4. 확정된 스냅샷으로 목록과 계획 부족량을 다시 계산한다.

총 필요량, 보유량, 부족량은 서로 다른 값이다. 스캔 데이터가 없는 상태를 보유량 0으로
표시해서도 안 된다. [@migration-baseline] v6의 인벤토리 화면도 단순 목록보다 계획과
연결된 부족 진단을 핵심 가치로 사용했다. [@v6-ui-audit]

### 사용자 검수 포인트

- 기본 진입을 `보유량 목록`과 `부족 진단` 중 어디에 둘지 정해야 한다.
- 계획 부족과 전체 육성 부족을 모두 유지할 경우 같은 흐름에 둘지 별도 기믹으로 나눌지
  정해야 한다.
- 스캔 비교를 인벤토리 탭에서도 완결할지, 스캔 탭의 검토 흐름으로 이동시킬지 정해야 한다.

## 7. 전술대항전 탭

### 기능 그룹

- **편성 작성**: 실제 보유 학생을 검색·필터해 공격 또는 방어 덱을 구성한다.
- **전적 기록**: 상대, 결과, 시즌·날짜, 덱과 메모를 저장한다.
- **기록 탐색·재사용**: 과거 기록을 검색하고 덱을 복사해 새 기록이나 편성의 시작점으로 쓴다.
- **족보·상대 자료 조회**: 상대나 방어덱 조건으로 기존에 저장한 대응 기록을 찾는다.

### 예상 주 흐름 A — 새 전적 기록

1. 시즌·날짜와 공격/방어 기록 종류를 고른다.
2. 내 덱과 상대 덱을 실제 학생 목록에서 구성한다.
3. 승패와 메모를 입력한다.
4. 저장 전 현재 편집 대상과 기록 종류를 확인한다.
5. 저장한 기록을 목록에서 확인하고 필요하면 수정한다.

### 예상 주 흐름 B — 과거 기록 재사용

1. 상대, 학생, 결과, 기간 등의 조건으로 기록을 검색한다.
2. 기록 상세와 당시 덱·메모를 확인한다.
3. 참고만 하거나 공격/방어 덱을 새 편성으로 복사한다.
4. 복사본을 수정해 새 기록으로 저장하며 원본은 유지한다.

### 예상 주 흐름 C — 족보 조회

1. 상대 또는 방어 덱 조건을 입력한다.
2. 저장된 족보와 관련 전적을 찾는다.
3. 적용할 대응 덱을 선택해 편성으로 보낸다.
4. 실제 결과를 새 전적으로 남겨 자료를 축적한다.

P6 범위는 기록·메모·검색·필터와 실제 학생 기반 편성까지다. 고급 추천과 승률 분석은
별도 범위이므로 이 흐름에 포함하지 않는다. [@p6-workflow]

### 사용자 검수 포인트

- 첫 화면의 주 행동이 `새 기록`, `편성`, `기록 검색` 중 무엇인지 정해야 한다.
- 공격 기록, 방어 기록과 족보를 별도 모드로 유지할지 하나의 기록 모델로 보여줄지
  확인해야 한다.
- 캡처·붙여넣기·폴더·Excel 같은 v6 입력 경로를 P6에서 어느 수준까지 복원할지 별도
  범위 결정이 필요하다. [@v6-ui-audit]

## 8. 통계 탭

### 기능 그룹

- **학생 성장 분포**: 보유·미보유, 성장 단계와 학생 메타데이터 기준으로 분포를 본다.
- **인벤토리 현황**: 카테고리별 보유 현황과 결측·오래된 데이터 상태를 본다.
- **계획 집계**: 계획의 총 필요량과 부족 재화 분포, 영향을 받는 학생 수를 본다.
- **상세 대상으로 이동**: 선택한 지표의 원인이 되는 학생, 재화 또는 계획 범위를 연다.

### 공통 예상 흐름

1. 학생 성장, 인벤토리, 계획 중 분석 주제를 고른다.
2. 프로필과 집계 범위를 확인한다.
3. KPI 또는 분포에서 이상·부족·관심 구간을 선택한다.
4. 해당 구간을 만든 대상 목록과 집계 기준을 확인한다.
5. 수정 행동이 필요하면 학생, 인벤토리 또는 계획 탭으로 이동한다.

통계는 결과를 직접 고치는 화면이 아니라 `관심 구간 발견 → 근거 확인 → 원본 데이터를
소유한 탭으로 이동`하는 흐름을 갖는다. 정렬은 표시만 바꾸며 집계 범위를 바꾸지 않는다.

### 사용자 검수 포인트

- 학생 통계의 범위를 프로필 전체로 할지, 학생 탭의 현재 필터 결과와 연동할지 확정해야
  한다. v6의 숨겨진 통계는 학생 탭 필터 결과 기반이었다. [@v6-ui-audit]
- 기본으로 보여 줄 KPI와 차트 목록은 P6 구현 전에 별도로 고정해야 한다. [@p6-workflow]
- 세 분석 주제를 한 화면의 요약으로 시작할지 명확한 하위 모드로 분리할지 정해야 한다.

## 9. 스캔 탭

### 기능 그룹

- **스캔 준비 확인**: 현재 프로필, 대상 게임 창, backend·scanner 상태와 실행 가능 여부를 본다.
- **스캔 실행 제어**: 학생 또는 인벤토리 등 종류를 선택하고 시작·진행 확인·취소·재시도한다.
- **후보 검토·확정**: 기존 확정값과 후보값, confidence와 근거를 비교해 수정·확정한다.
- **최근 결과 확인**: 최근 세션의 성공·실패·취소·검토 대기 결과와 진단을 다시 연다.

### 예상 주 흐름 A — 새 스캔

1. 현재 프로필과 대상 창이 맞는지 확인한다.
2. 연결과 화면 조건이 충족되지 않으면 설정 또는 안내된 복구 행동으로 이동한다.
3. 스캔 종류와 필요한 범위를 선택한다.
4. 스캔을 시작하고 phase, 진행률과 현재 인식 대상을 확인한다.
5. 필요하면 취소하거나 실패 원인을 확인한 뒤 재시도한다.
6. 완료된 후보를 검토한다.
7. 수정·확정된 결과만 repository에 반영한다.
8. 학생 또는 인벤토리 탭으로 이동해 반영 결과를 확인한다.

### 예상 주 흐름 B — 최근 결과 복구

1. 최근 세션에서 실패 또는 검토 대기 항목을 선택한다.
2. 실패면 재시도 가능한 조건과 진단을 확인한다.
3. 검토 대기면 기존값과 후보값을 비교한다.
4. 재시도, 수정·확정 또는 폐기를 선택한다.

스캔 시작, session generation, 취소, stale event 무시, 후보 검토와 repository commit은
분리된 상태여야 한다. 낮은 confidence 결과를 자동 확정하지 않는다. [@p6-workflow]

### 사용자 검수 포인트

- 스캔 종류를 사용 빈도 기준으로 단순화할지, 세부 모드를 처음부터 모두 노출할지 정해야 한다.
- 성공한 높은 confidence 항목도 전체 검토를 거칠지, 변경 요약만 확인하고 확정할지 정해야 한다.
- 일반 상태 메시지와 기술 로그·진단을 같은 흐름에 둘지 진단 기믹으로 분리할지 정해야 한다.

## 10. 설정 탭

### 기능 그룹

- **프로필 관리**: 프로필을 생성·선택·이름 변경하고 현재 적용 상태를 확인한다.
- **게임 창·scanner 준비**: 대상 창을 선택하고 scanner가 사용할 연결 상태를 확인한다.
- **backend 연결 관리**: 재연결·재시작, protocol/version과 경로 상태를 확인한다.
- **진단·로그**: backend와 scanner 오류, 로그 위치와 환경 정보를 확인한다.
- **그래픽 진단 진입**: Adaptive-Sync 진단 화면을 연다.
- **위험 작업**: 현재 프로필 데이터 삭제처럼 되돌리기 어려운 작업을 명시적으로 확인한다.

### 예상 주 흐름 A — 프로필 시작·전환

1. 기존 프로필을 선택하거나 새 프로필을 만든다.
2. 적용될 프로필과 데이터 범위를 확인한다.
3. 프로필을 적용한다.
4. 해당 프로필의 연결·데이터 상태가 다시 로드됐는지 확인한다.
5. 데이터가 없으면 게임 창을 선택하고 스캔 탭으로 이동한다.

### 예상 주 흐름 B — 연결 복구

1. 홈, 스캔 또는 현재 탭의 disconnected 상태에서 설정으로 이동한다.
2. backend, protocol/version, 경로, 대상 창 중 실패 지점을 확인한다.
3. 적합한 재연결·재시작·창 재선택을 수행한다.
4. 준비 상태가 복구됐는지 확인하고 원래 작업으로 돌아간다.

### 예상 주 흐름 C — 진단과 위험 작업

1. 일반 설정과 분리된 진단 또는 데이터 관리 기능을 연다.
2. 로그·scanner·그래픽 진단 중 대상을 고른다.
3. 결과를 확인하거나 필요한 정보를 복사한다.
4. 프로필 삭제는 대상과 영향을 다시 확인한 뒤 명시적으로 확정한다.

### 사용자 검수 포인트

- 설정의 기본 초점을 `프로필`, `게임 창`, `backend 상태` 중 어디에 둘지 정해야 한다.
- 일반 사용자의 복구 행동과 개발자용 protocol·경로·로그 진단을 별도 기믹으로 분리할지
  정해야 한다.
- 프로필 삭제 외에 백업·복원·import를 P6에 넣을지는 별도 범위 결정이 필요하다.

## 11. Adaptive-Sync 진단 화면

Adaptive-Sync 진단은 기본 탭이 아니라 설정에서 진입하는 보조 흐름이다. [@p6-workflow]

1. 사용자가 그래픽 표시 문제를 겪고 설정에서 진단을 연다.
2. 재현 조건과 현재 표시 환경을 확인한다.
3. PNG hover, 탭 전환, 창 크기 변경 등 안내된 재현 동작을 수행한다.
4. 결과를 확인하거나 진단 정보를 복사한다.
5. 설정 또는 원래 탭으로 돌아간다.

진단 화면은 일반 작업 요약이나 다른 탭의 기능을 가져오지 않는다.

## 12. 탭을 가로지르는 핵심 여정

개별 탭의 섹션은 다음 전체 여정 중 자신이 소유하는 단계만 담당한다.

1. **설정**에서 프로필과 게임 창을 준비한다.
2. **스캔**에서 현재 학생·인벤토리 후보를 얻고 검토·확정한다.
3. **학생**에서 현재 상태를 확인하고 대상 학생을 계획에 추가한다.
4. **계획**에서 목표와 프리셋을 정하고 총 필요량을 계산한다.
5. **인벤토리**에서 보유량과 비교한 부족량과 영향 학생을 판단한다.
6. **통계**에서 여러 학생·재화·계획에 걸친 분포를 보고 상세 탭으로 이동한다.
7. **홈**은 이 여정의 현재 상태와 다음 재개 지점을 요약한다.

전술대항전은 동일한 프로필의 실제 학생 데이터를 사용하지만 성장 계획 여정과는 독립된
기록·재사용 흐름으로 취급한다.

## 13. 구현 전 갱신 규칙

- 사용자가 흐름을 승인하면 해당 탭의 `사용자 검수 포인트`를 결정문으로 바꾸고 흐름의
  분기와 완료 상태를 확정한다.
- 그 뒤에만 기능 그룹을 실제 사선 섹션으로 매핑한다.
- 섹션 구성 문서는 기능 그룹마다 부착면, 주 모션, 내부 행과 좁은 화면 대체 흐름을
  기록해야 한다.
- 구현 중 코드와 이 문서의 사용자 흐름이 달라지면 코드에 맞춰 조용히 바꾸지 않고
  사용자 결정 또는 워크플로 범위 변경을 먼저 기록한다.
- P6 단계 상태, 산출물과 검증 결과는
  [P0-P6 Workflow Status](../workflows/p0-p6-workflow-status)에 계속 기록한다.
- 창 크기와 비율에 따라 섹션을 재배치할 때의 허용 범위, 사선 안전 폭과 레이아웃 상태
  선택 규칙은 [Responsive Diagonal Layout Policy](responsive-diagonal-layout-policy)의
  검수 전 제안을 함께 사용한다.
