# 현재 UI 감사

## 1. 문서 목적과 범위

이 문서는 BA Planner의 현재 사용자 경험을 재설계하기 전에 작성한 현황 감사다. 목표는 기존 기능과 데이터 의미를 바꾸지 않은 채, 사용자가 실제로 마주치는 화면 구조와 정보 우선순위, 시각 체계, 해상도 위험을 정리하는 것이다. 이후 ImageGen 콘셉트 이미지와 인터랙티브 목업을 만들 때의 기준 문서로 사용할 수 있도록 구성했다.

이번 조사에서는 제품 UI 코드를 수정하지 않았다. 현재 노출되는 Qt 플래너, 스캔 중 별도 실행되는 Tk 오버레이와 보조 대화상자, 관련 상태·저장 흐름을 코드와 프로젝트 문서에서 확인했다. 저장된 프로필 데이터와 과거 오류 제보 이미지는 제품 구조의 근거로 사용하지 않았고, 이미지는 화면 분위기와 과거 표시 문제를 보조 확인하는 용도로만 보았다.

중요한 범위 구분은 다음과 같다.

- 현재 기본 Qt 탭: `스캔`, `학생`, `플랜`, `필요 재화`, `인벤토리`, `전술대항전`, `설정`
- 구현되어 있지만 현재 기능 플래그로 숨겨진 탭: `공략 타임라인`, `Statistics`
- 별도 Tk 표면: 상태별 플로팅 오버레이, 창 선택, 빠른 학생 스캔, 빠른 스캔 기준 목록 편집, 스캔 결과 검토, 입력 테스트
- 이 문서의 평가는 코드 구조가 아니라 사용자의 목표 달성 과정과 화면에서 보이는 정보의 관계를 기준으로 한다.

실사용 빈도에 관한 텔레메트리는 코드베이스에 없으므로, 아래의 “자주 수행하는 작업”은 앱의 기본 진입 흐름, 반복 노출되는 액션, 저장 모델을 바탕으로 한 제품적 추정이다.

## 2. 현재 제품의 사용자 모델

BA Planner는 하나의 단순한 뷰어라기보다 다음 네 가지 일을 한 제품 안에서 수행한다.

1. 게임 창을 연결하고 현재 상태를 스캔한다.
2. 스캔된 학생과 인벤토리를 탐색한다.
3. 학생별 성장 목표를 만들고 필요한 재화를 계산한다.
4. 인벤토리 부족과 전술대항전 기록을 의사결정 자료로 사용한다.

핵심 가치는 “직접 입력을 줄이고, 현재 상태에서 목표까지 무엇이 필요한지 보여주는 것”이다. 따라서 재설계 시에도 장식적인 대시보드보다 `스캔 상태의 신뢰성`, `현재값과 목표값의 구분`, `필요량과 보유량과 부족량의 구분`, `학생을 찾고 계획에 넣는 속도`가 우선되어야 한다.

## 3. 주요 사용자 흐름

### 3.1 최초 또는 프로필 전환 후 시작

1. 사용자가 프로필을 선택하거나 생성한다.
2. 설정에서 실행 중인 Blue Archive 창을 선택한다.
3. 스캔 탭 또는 플로팅 오버레이에서 스캔 종류를 고른다.
4. 스캔 중 진행 상태와 현재 인식 대상을 확인한다.
5. 필요한 경우 검토 대화상자에서 의심 수량을 확인하고 저장한다.
6. 저장 결과가 프로필의 현재 학생·인벤토리 상태에 반영되고 Qt 화면이 다시 읽는다.

이 흐름의 핵심 질문은 항상 “지금 어떤 프로필과 어떤 게임 창에 쓰고 있는가?”다. 현재 화면은 이 정보를 텍스트로 제공하지만, 프로필·창 연결·스캔 가능 상태가 하나의 명확한 준비 상태로 묶이지는 않는다.

### 3.2 학생 확인에서 계획 추가까지

1. 학생 탭에서 이름·ID·태그 검색, 정렬, 보유/서버 필터를 사용한다.
2. 카드 그리드에서 학생을 선택한다.
3. 우측 상세에서 현재 레벨, 성급, 전용무기, 스킬, 장비, 전투·인연 스탯을 확인한다.
4. `플랜에 추가`로 계획에 넣는다.
5. 플랜 탭에서 학생을 선택하고 목표 성급·무기·스킬·장비·레벨·능력개방 값을 편집한다.
6. 선택 학생의 필요 재화 미리보기로 결과를 확인한다.

이 제품의 가장 대표적인 주 흐름이다. 현재는 학생 탐색과 계획 편집이 서로 다른 탭에 있고, `학생 탭에서 보기`와 `플랜에 추가`로 왕복을 지원한다.

### 3.3 여러 학생의 필요 재화 계산

1. 필요 재화 탭에서 계산 범위와 검색 모드를 오간다.
2. 검색·필터된 학생 중 일부 또는 전체를 범위에 추가한다.
3. 계획이 없는 학생에 대해 레벨·장비·스킬 기본 계산 포함 여부를 선택한다.
4. 우측 합산 결과에서 필요량, 보유량, 부족량을 확인한다.
5. 일반 순서 또는 부족 비율 순서로 결과를 정렬한다.

기능적으로 강력하지만, `학생 필터`, `검색 결과 선택`, `계산 범위`, `계획 유무`, `미계획 학생 계산 옵션`이 한 화면에 함께 있어 처음 쓰는 사용자가 범위의 의미를 이해하기 어렵다.

### 3.4 인벤토리 부족 분석

1. 장비 또는 아이템 모드를 선택한다.
2. 카테고리/시리즈 하위 탭과 정렬을 사용해 항목을 찾는다.
3. 좌측에서 계획 부족 TOP 5, 전체 육성 부족 TOP 5, 충족도와 학교별 위험을 본다.
4. 중앙 목록에서 재화를 고른다.
5. 우측 상세에서 보유, 계획 필요·부족, 전체 육성 필요·부족, 영향 학생과 판단 힌트를 확인한다.

단순 보유 목록이 아니라 계획과 연결된 진단 화면이라는 점이 차별점이다. 반면 하나의 화면에 “목록 탐색”과 “분석 대시보드”가 동등한 밀도로 공존해 초점이 쉽게 분산된다.

### 3.5 전술대항전 기록과 재사용

1. 날짜·시즌과 공격/방어/족보 모드를 고른다.
2. 상대, 결과, 덱, 메모를 직접 입력하거나 캡처·붙여넣기·폴더·최근 기록으로 채운다.
3. 전적을 저장하고 중앙 기록 목록에서 검색한다.
4. 선택 기록의 ATK/DEF 덱을 복사·수정·삭제한다.
5. 우측에서 상대별 기록이나 방어덱에 맞는 족보를 검색한다.

반복 작업을 빠르게 만드는 도구가 많지만, 세 열 모두 입력과 액션이 많아 신규 사용자는 어디서 시작해야 하는지 판단하기 어렵다.

## 4. 가장 자주 수행할 것으로 보이는 작업

우선순위는 다음과 같이 추정된다.

1. 게임 창 연결 상태 확인과 학생/아이템/장비 스캔
2. 학생 검색 후 현재 성장 상태 확인
3. 학생을 플랜에 추가하고 목표값 편집
4. 계획 전체의 필요량·보유량·부족량 확인
5. 부족 재화와 영향을 받는 학생 확인
6. 프로필 전환과 데이터 새로고침
7. 전술대항전 전적 입력·검색·덱 복사

현재 최상위 탭은 이 빈도 차이를 표현하지 않고 모두 같은 시각적 비중으로 나열한다. `스캔 → 학생 → 플랜 → 필요 재화 → 인벤토리`는 하나의 연속된 성장 계획 흐름이지만, 내비게이션에서는 독립 도구처럼 보인다. `전술대항전`은 성격이 다른 기록 도구인데 동일한 레벨에 놓여 있다.

숨겨진 `Statistics` 탭은 현재 사용자 경험 범위에서는 제외해야 한다. 향후 다시 노출할 경우, 학생 탭의 현재 필터 결과를 기반으로 한다는 의미가 화면에 명확히 드러나야 한다. 현재 코드에는 단순 KPI/분포를 넘어 선버스트와 상세 분석 UI가 존재하므로, 재노출 전에 별도 범위 정의가 필요하다.

## 6. 현재 레이아웃 구조

### 6.1 전체 Qt 플래너

- 1920×1080을 설계 기준으로 하며 현재 모니터의 16:9 작업 영역에 맞춘 뒤 창 크기를 고정한다.
- 바깥 여백 안에 상단 가로 탭 바와 현재 탭 내용이 놓인다.
- 탭 내용은 대부분 `화면 헤더 → 도구/상태 패널 → 본문`의 수직 구조다.
- 본문은 화면 성격에 따라 2열 또는 3열 스플리터를 사용한다.
- 작은 화면에서는 전체 UI 스케일을 줄이고, 매우 작은 16:9 화면에서는 추가 축소한다.

### 6.2 반복되는 화면 패턴

- 화면 헤더: 큰 제목, 한두 줄 설명 또는 상태, 우측 수량 요약
- 표준 패널: 둥근 모서리의 어두운 면, 옅은 테두리
- 카드 그리드: 평행사변형 학생 초상 카드와 반응형 열 수
- 마스터-디테일: 좌측 목록, 우측 고정 폭 상세
- 작업공간: 좌측 대상 목록, 우측 편집기 또는 결과
- 분석 화면: 좌측 요약, 중앙 목록, 우측 상세
- 상태/보조 문구: 연한 회색 텍스트와 `filterSummary` 형태의 요약 줄
- 내부 모드: 상위 탭 안에서 다시 버튼, 하위 탭, 스택을 사용

### 6.3 화면별 구조

- 스캔: 상단 연결/액션 헤더, 좌측 보고 로그와 실시간 인식 카드, 우측 진행 제어와 16:9 미리보기의 비대칭 2열 구조
- 학생: 헤더, 한 줄 도구막대, 필터 요약, 넓은 카드 그리드와 356~408px 범위의 우측 상세 패널
- 플랜: 헤더, 빠른 추가 밴드, 좌측 계획 카드 목록과 우측 스크롤 편집기. 편집기는 다시 `목표 편집/필요 재화` 모드로 나뉨
- 필요 재화: 공용 검색 도구막대, 좌측 `범위/검색` 모드, 우측 합산 결과의 2열 구조
- 인벤토리: 헤더, 장비/아이템 모드와 다수의 하위 카테고리, 좌측 진단 요약·중앙 목록·우측 상세의 3영역 구조
- 전술대항전: 입력, 기록, 상대/족보 인사이트를 동시에 보여주는 3열 구조
- 설정: 헤더 아래 계정 관리 패널과 창 인식 패널을 세로 배치

## 7. 공유 UI 구성요소

재설계 시 기능 연결을 유지하면서 외형을 바꾸기 좋은 공통 단위는 다음과 같다.

- 메인 탭 바와 하위 탭 바
- 화면 헤더, 표준 패널, 섹션 패널, 한 줄 밴드
- 학생 카드, 학생 초상, 미보유 배지, 상세 성장 스트립
- 반응형 카드 그리드와 둥근 리스트
- 평행사변형 패널과 평행사변형 버튼
- 검색 입력, 정렬 드롭다운, 체크박스, 필터 대화상자
- 목표 구간 선택기, 숫자 스테퍼, 두 자리 선택기
- 필요 재화 칩, 인벤토리 행, 충족률/부족률 바
- 스크롤 영역과 커스텀 스크롤 핸들
- 전술 덱 편집기와 학생 슬롯
- 빈 상태, 상태 요약, 진행 중 오버레이

현재 공유 단위가 이미 충분히 많아, 목업은 개별 화면을 제각각 그리기보다 `내비게이션`, `페이지 헤더`, `검색/필터 도구막대`, `마스터-디테일`, `재화 행/칩`, `목표 입력 행`, `상태 메시지`를 디자인 시스템 단위로 먼저 정의하는 편이 적합하다.

## 8. 현재 UI의 장점

### 8.1 기능과 정보 연결

- 학생 상세에서 바로 플랜에 추가하고, 플랜에서 다시 학생 탭으로 돌아갈 수 있어 핵심 왕복 흐름이 존재한다.
- 플랜 편집 중 선택 학생의 필요 재화를 즉시 미리 볼 수 있다.
- 필요 재화와 인벤토리 화면이 단순 총량이 아니라 보유량과 부족량까지 연결한다.
- 인벤토리는 계획 부족과 전체 육성 압력을 분리해 단기 목표와 장기 목표를 함께 판단하게 한다.
- 전술대항전은 수동 입력, 최근 기록, 캡처, 붙여넣기, 폴더, Excel 등 다양한 입력 경로를 지원한다.

### 8.2 상태 가시성

- 스캔 화면에서 프로필, 대상 창, 연결 여부, 진행률, 중지, 현재 인식 대상과 로그를 제공한다.
- 학생 수, 보유 수, 필터 적용 여부, 계획 학생 수, 계산 범위 수 등 대부분의 화면에 현재 범위 요약이 있다.
- 미보유 학생, 부족 재화, 스캔 검토 대상처럼 주의가 필요한 상태를 배지·색·강조 행으로 구분한다.
- 저장 파일 변경을 감지해 화면 데이터를 다시 읽으므로 별도 스캔 프로세스와 뷰어 간 결과 반영이 자연스럽다.

### 8.3 시각적 정체성

- 어두운 남색 바탕, 분홍 포인트, 학생 초상과 평행사변형 카드가 게임 팬 도구로서 뚜렷한 인상을 만든다.
- 학교, 공격 타입, 방어 타입, 성급, 장비 티어 등에 의미 색을 사용해 빠른 패턴 인식에 도움이 된다.
- 화면 대부분이 동일한 둥근 패널, 여백, 테두리 언어를 사용해 기본적인 일관성이 있다.
- 한국어 제목용 폰트를 제품 전체에 적용하고 있어 분위기가 통일되어 있다.

### 8.4 대량 데이터 탐색

- 검색, 정렬, 보유 포함, JP 전용 제외, 메타데이터 다중 필터를 조합할 수 있다.
- 학생 카드는 가시 영역의 이미지를 우선 로딩하고 그리드 열 수를 조정한다.
- 필요 재화 범위에 현재 필터 결과 전체 또는 계획 학생만 빠르게 추가할 수 있다.
- 긴 목록과 상세 편집기는 필요한 곳에 스크롤을 사용한다.

## 9. 현재 UI의 문제점

### 9.1 제품 구조와 내비게이션

- 최상위 탭이 7개이며 모든 탭이 같은 무게로 보인다. 성장 계획의 연속 단계와 독립 도구인 전술대항전·설정의 차이가 드러나지 않는다.
- 스캔은 Qt 탭과 Tk 플로팅 오버레이 두 곳에서 시작할 수 있다. 상황에 맞는 진입점이라는 장점은 있지만, 사용자는 어느 쪽이 “주 화면”인지 학습해야 한다.
- 화면마다 큰 헤더가 반복되어 현재 위치를 재확인하는 데 비해 상당한 세로 공간을 사용한다.
- `Blue Archive Planner`, `계획 작업공간`, `필요 재화량`, `인벤토리 상태 분석`처럼 제목의 명명 수준과 언어가 일정하지 않다.
- 숨겨진 탭 코드와 과거 화면 자료에는 `Statistics` 같은 영어 탭이 섞여 있어 향후 노출 시 언어 정책을 다시 정해야 한다.

### 9.2 행동 우선순위

- 기본 버튼이 대부분 투명 배경과 분홍 외곽선을 사용한다. 주요 행동, 보조 행동, 위험 행동의 차이가 약하다.
- 스캔 탭의 학생·현재 학생·자원·아이템·장비 버튼이 같은 위계라서 일반 사용자가 어떤 순서로 시작해야 하는지 알기 어렵다.
- 전술대항전의 `전적 추가`, `새 입력`, `캡처`, `Folder`, `붙여넣기`, `최근 공격`, `최근 방어`, `ATK Copy`, `DEF Copy`, `Excel Import`가 한 화면에서 경쟁한다.
- 설정의 `현재 프로필 데이터 삭제`가 파괴적 작업임에도 일반 버튼 계열에 가까워 보인다.
- 플랜 편집은 값 변경 즉시 저장되지만 화면에서 자동 저장 여부와 마지막 저장 상태가 두드러지지 않는다.

### 9.3 중복과 인지 부하

- 학생 탭과 필요 재화 탭에 거의 동일한 검색·정렬·미보유·JP 전용·필터·새로고침 도구막대가 반복된다.
- 필요 재화 탭에서 “검색 결과에 보임”, “추가 대기”, “계산 범위에 포함”, “계획에 포함”의 네 상태가 생길 수 있어 카드 상태를 오해하기 쉽다.
- 인벤토리는 상위 장비/아이템 모드, 다수의 하위 카테고리 탭, 정렬, 진단 카드, 목록, 상세를 동시에 보여준다.
- 플랜은 빠른 추가, 계획 카드 목록, 편집기, 필요 재화 미리보기를 모두 한 화면에 유지하여 초점이 분산된다.
- 전술대항전은 입력·기록·분석 세 작업을 동시에 노출해 숙련자에게는 빠르지만 신규 사용자에게는 진입 장벽이 높다.

### 9.4 빈 상태와 안내

- 여러 화면이 설명 문장을 제공하지만 “다음에 누를 버튼”과 직접 연결되는 단계형 안내는 약하다.
- 스캔 미실행, 계획 없음, 범위 없음, 인벤토리 없음 상태마다 문구는 있으나 제품 전체에서 같은 빈 상태 문법을 사용하지 않는다.
- 미보유 학생의 현재값 없음, 계획 목표 없음, 스캔값 없음이 시각적으로 비슷한 `-` 또는 빈 영역으로 표현될 수 있다.
- 스캔의 상세 로그는 신뢰성 확인에는 유용하지만 일반 사용자의 완료 여부 판단과 기술 진단 정보가 같은 공간에 있다.

### 9.5 상호작용 발견 가능성

- 학생 카드와 계획 카드의 선택 상태, 필요 재화 검색 카드의 대기 선택 상태, 인벤토리 행 선택이 서로 다른 목적이지만 유사한 카드/테두리 언어를 쓴다.
- 스플리터 핸들은 투명하며 폭만 존재해 사용자가 열 너비를 조절할 수 있다는 단서가 약하다.
- 계획 학생 순서를 드래그해 인벤토리 우선순위에 반영하는 기능은 보조 문구에 의존한다.
- 커스텀 구간 선택기는 많은 값을 빠르게 고를 수 있지만, 현재값·목표값·선택 가능 범위의 의미를 처음 보는 사용자가 즉시 이해하기 어렵다.
- 플로팅 오버레이는 앱 상태와 로비 여부에 따라 버튼 구성이 크게 바뀌므로, 사라진 기능을 오류로 오인할 가능성이 있다.

## 10. 정보 계층 문제

### 10.1 현재값, 목표값, 결과값

학생 상세의 현재값, 플랜 편집의 목표값, 필요 재화의 계산 결과는 데이터 의미가 다르지만 화면을 탭 단위로 나누면서 관계를 사용자가 기억해야 한다. 재설계에서는 같은 학생 맥락 안에서 다음 세 층을 반복적으로 구분해야 한다.

- 현재: 스캔으로 확인한 실제 보유 상태
- 목표: 사용자가 설정한 성장 계획
- 차이: 현재에서 목표까지 필요한 비용

색만으로 구분하지 말고 라벨, 위치, 문장형 요약도 함께 사용해야 한다.

### 10.2 필요, 보유, 부족

인벤토리와 필요 재화 화면은 `필요`, `보유`, `부족`을 실제로 분리해 계산하지만, 한 행 또는 칩 안에 숫자가 여러 개 들어가면서 첫 시선이 어디에 가야 하는지 불명확해질 수 있다. 일반적인 의사결정 순서는 `부족 여부 → 부족량 → 필요량/보유량 근거 → 영향 학생`이다. 현재 화면은 데이터 열을 나란히 보여주는 데 강하지만 이 순서를 충분히 강조하지 않는다.

또한 `계획 부족`과 `전체 육성 부족`은 서로 다른 범위인데 유사한 형태로 함께 노출된다. 범위 배지나 섹션 배경으로 명확히 분리할 필요가 있다.

### 10.3 상태와 설명의 경쟁

화면 헤더 설명, 필터 요약, 패널 제목, 상세 메타, 빈 상태가 모두 옅은 회색 작은 글씨를 사용한다. 중요한 연결 경고나 범위 설명도 일반 보조문과 비슷해 놓치기 쉽다. 특히 다음은 일반 설명보다 높은 위계가 필요하다.

- 선택된 게임 창이 없거나 비율이 맞지 않음
- 현재 계산이 계획 목표를 쓰는지, 미계획 기본 목표를 쓰는지
- 인벤토리 스캔이 없어 보유/부족 계산이 불완전함
- 스캔 결과가 검토를 기다림
- 자동 저장 성공/실패

### 10.4 전술대항전의 작업 맥락

입력 중인 기록, 목록에서 선택한 과거 기록, 상대 검색 결과, 족보 검색 결과가 한 화면에 공존한다. “현재 편집 대상”과 “참고 중인 기록”이 분리되지 않으면 잘못된 기록을 수정하거나 복사할 위험이 있다. 선택한 기록을 편집 모드로 불러왔을 때는 화면 전반에 일관된 편집 상태 표시가 필요하다.

## 11. 타이포그래피 문제

- 기본 UI 폰트는 경기천년제목 Medium이며 11pt를 기준으로 한다. 개성이 강한 제목용 서체를 본문, 입력, 수치, 긴 설명까지 넓게 쓰면 장시간 정보 탐색에서 피로가 생길 수 있다.
- 제목 24px, 상세 이름 24~28px, 큰 레벨 44px처럼 강조 크기는 충분하지만 중간 계층 대부분이 10~15px 범위에 몰려 있다.
- `count`, `detailSub`, `filterSummary`, `sectionSub`, `kpiValueSub`가 같은 muted 스타일을 공유해 설명, 범위, 메타, 상태의 역할 차이가 사라진다.
- `LEVEL`, `EX`, `N`, `P`, `S`, `ATK Copy`, `DEF Copy`, `Folder`, `Excel Import` 등 영어와 약어가 한국어 UI에 혼재한다.
- 작은 화면에서는 전체가 비례 축소되므로 10~11px 기준 텍스트가 더 작아진다. 고밀도 인벤토리 행과 전술 덱 영역에서 가독성 문제가 가장 먼저 발생할 가능성이 높다.
- 수치 정보는 자릿수 비교가 중요하지만 별도의 숫자용 서체나 tabular number 정책이 보이지 않는다.

## 12. 색상 문제

현재 핵심 팔레트는 분홍 `#F266B3`, 밝은 연보라 `#EFE4F2`, 패널 남색 `#313B59`, 배경 남색 `#2C3140`, 밝은 텍스트 `#F2F2F2`를 기반으로 한다.

- 브랜드 인상은 명확하지만 분홍 외곽선이 탭, 일반 버튼, 선택 버튼, 작은 모드 버튼에 반복되어 모든 인터랙션이 같은 중요도로 보인다.
- 보조 텍스트는 밝은 텍스트와 패널색을 혼합한 muted 색을 광범위하게 사용한다. 작은 글씨와 얇은 폰트 조합에서는 대비가 부족할 수 있다.
- 공격 타입, 방어 타입, 학교, 성급, 장비 티어, 계획 부족, 전체 육성 압력 등 많은 의미 색이 동시에 사용된다. 범례가 없으면 색의 의미를 기억해야 한다.
- 빨강은 부족, 오류, 삭제 등 여러 부정 의미를 맡는다. 수치 부족과 파괴적 액션을 형태·아이콘·문구로 추가 구분해야 한다.
- 선택 상태가 주로 분홍 테두리 굵기 변화에 의존하므로 색각 차이와 저해상도 렌더링에서 구분력이 떨어질 수 있다.
- 실제 게임 화면의 밝은 하늘색·흰색 위에 어두운 Tk 대화상자나 오버레이가 뜰 때는 가독성은 높지만, Qt 플래너와 Tk 표면의 색·모서리·타이포그래피가 완전히 같은 제품처럼 느껴지지는 않는다.

## 13. 간격과 밀도 문제

- 최상위 바깥 여백 16px, 화면 간격 12px, 패널 내부 14~18px로 기본 리듬은 일관적이다.
- 그러나 헤더와 도구막대가 각각 큰 패널을 차지해 본문 시작 전 세로 공간이 많이 소모된다.
- 학생 상세는 초상, 성장 스트립, 이름/배지, 타입 칩, 플랜 버튼, 레벨/포지션/클래스/무기, 스킬, 장비, 스탯이 한 세로 열에 쌓인다. 개별 간격은 규칙적이지만 전체적으로 긴 “정보 벽”이 된다.
- 플랜 편집은 각 목표를 밴드형 행으로 감싸고 내부에도 선택 셀을 반복해 테두리와 여백이 누적된다.
- 인벤토리 좌측에는 230px 고정 높이 패널 두 개와 126px 패널 등이 쌓여, 작업 영역 높이에 따라 아래 정보가 압축되거나 스크롤 없이 잘릴 위험이 있다.
- 전술대항전의 버튼 행은 버튼 수가 많고 일부는 고정 폭이다. 언어 변경이나 DPI 변화에서 텍스트가 잘리거나 지나치게 조밀해질 수 있다.
- 플로팅 오버레이는 폭이 좁고 최근 로그를 3줄만 보여주므로 긴 오류 메시지와 여러 상태가 압축된다.

## 14. 패널 구조 문제

- 거의 모든 정보가 둥근 패널 또는 밴드에 들어가 있어 그룹화는 명확하지만, 패널이 많아질수록 어느 경계가 중요한지 약해진다.
- 상위 패널 안에 하위 패널, 모드 버튼, 하위 탭, 다시 리스트 카드가 들어가는 중첩이 특히 인벤토리와 플랜에서 깊다.
- 배경색 변화가 미세하고 테두리도 옅어, 패널 중첩이 구조를 설명하기보다 화면을 복잡하게 만들 수 있다.
- 학생 카드의 평행사변형 모티프는 대표성이 높지만, 동일 모티프를 일반 버튼·상태 패널까지 확장하면 콘텐츠보다 장식 윤곽이 먼저 보인다.
- 마스터-디테일 패턴은 적절하지만, 현재 상세 패널이 항상 공간을 차지해 학생을 빠르게 훑는 작업에서는 그리드 밀도를 낮춘다.

## 15. 해상도와 디스플레이 위험

### 15.1 현재 대응 방식

- 기준 해상도는 1920×1080이며 작업 영역의 폭과 높이에 맞춰 전체 UI 스케일을 계산한다.
- 16:9가 아닌 화면에서도 16:9 영역을 맞추고 남는 영역을 제외한다.
- 계산된 스케일이 0.85보다 작으면 0.9를 추가 적용해 작은 16:9 화면에서 더 축소한다.
- 창은 계산된 작업 영역 크기로 고정되어 사용자가 임의로 줄이거나 키울 수 없다.

### 15.2 예상 문제

- 1366×768급 화면에서는 대략 0.64 수준으로 축소될 수 있다. 10~15px 보조 텍스트, 34px 입력, 50~64px 목록 행이 지나치게 작아질 가능성이 높다.
- 1536×864에서는 축소는 덜하지만 3열 전술대항전과 인벤토리의 고정 폭·고정 높이 요소가 여전히 조밀하다.
- 16:10, 21:9, 세로형 또는 작업 표시줄 위치가 다른 환경에서는 가용 화면 전체를 쓰지 못하거나 비어 있는 공간이 생길 수 있다.
- 전체 창 고정 크기는 사용자가 보조 모니터, 화면 분할, 확대 도구와 함께 쓰기 어렵게 한다.
- 학생 상세의 최소/최대 폭, 스캔 미리보기 560px 최소 폭, 인벤토리 상세 320~360px 최소 폭, 전술 3열 초기 폭이 겹치면 작은 폭에서 본문이 압축된다.
- 고정 높이 패널과 세로 스크롤이 없는 상위 컨테이너는 낮은 작업 영역에서 하단 콘텐츠가 가려질 위험이 있다.
- 4K 고해상도에서는 최대 1.8배로 제한되어 물리적 크기와 OS 배율 조합에 따라 UI가 기대보다 작을 수 있다.
- Qt 화면은 모니터에 맞춰 고정되지만 Tk 보조 창은 별도 스케일 계산과 고정 대화상자 크기를 사용한다. 서로 다른 DPI의 다중 모니터를 오갈 때 크기와 선명도가 달라질 수 있다.
- 탭 바는 스크롤 버튼을 허용하므로 폭이 부족할 때 탭이 숨겨질 수 있지만, 숨겨진 탭의 존재를 사용자가 놓치기 쉽다.

### 15.3 재설계 검증 해상도

목업과 실제 구현 검증은 최소 다음 조합을 포함하는 것이 좋다.

- 1920×1080, Windows 100% 배율: 기준 화면
- 1536×864 또는 1920×1080, Windows 125% 배율: 일반 노트북
- 1366×768, Windows 100% 또는 125% 배율: 최소 지원 후보
- 2560×1440, Windows 125% 배율: 고해상도 데스크톱
- 3440×1440: 울트라와이드의 빈 공간 및 최대 폭 확인
- 서로 다른 DPI의 듀얼 모니터: Qt/Tk 표면 이동과 재배치 확인

## 16. UI 변경 시 절대 보존해야 할 기능과 데이터 흐름

### 16.1 데이터 의미의 다섯 구획

다음 값은 화면에서 가까이 보여도 같은 필드나 같은 계산으로 합치면 안 된다.

1. 스캔된 현재 학생 상태
2. 정적 학생 메타데이터
3. 사용자가 정한 플랜 목표
4. 현재에서 목표까지의 계산 결과
5. 계산 결과와 보유 인벤토리를 결합한 부족량

특히 UI 편의를 위해 목표값을 현재값에 덮어쓰거나, 필요량을 부족량으로 재정의하거나, 미보유 학생에게 가상의 현재 성장값을 채우면 안 된다.

### 16.2 스캔·저장 경계

- Tk 제어 앱은 대상 창 선택, 스캔 상태 전이, 검토와 저장 흐름을 소유한다.
- 실제 캡처, 입력, 인식, ROI와 화면 이동은 스캐너 계층의 책임이다.
- 스캔 결과의 병합과 현재 상태 저장은 repository 계층을 통과해야 한다.
- UI가 프로필 JSON 또는 SQLite를 직접 임의 수정하면 안 된다.
- 스캔 중지, 오류, 완료, 검토 취소의 상태 의미를 유지해야 한다.
- Qt에서 시작한 스캔은 별도 스캐너 프로세스와 상태 파일을 통해 진행 상황을 주고받는다. 목업에서 단일 프로세스처럼 보이게 만들더라도 이 통신 계약은 유지해야 한다.
- 대상 창과 프로필은 스캔 전에 명시적으로 확인 가능해야 한다.

### 16.3 학생·필터·통계

- 학생 UI의 기본 결합 모델은 현재 스캔 상태와 메타데이터를 합친 `StudentRecord` 의미를 유지해야 한다.
- 미보유 학생은 메타데이터 목록에는 존재하지만 현재 성장 상태는 없는 학생이다.
- 검색과 필터는 이름·ID·태그·메타데이터 값을 사용하며, 보유 표시와 JP 전용 숨김 의미를 유지해야 한다.
- 숨겨진 통계 탭을 다시 노출할 경우 통계는 전체 학생이 아니라 학생 탭의 현재 필터 결과를 기반으로 한다.
- 정렬은 표시 순서만 바꾸고 집계 대상이나 계산 의미를 바꾸면 안 된다.

### 16.4 플랜과 비용 계산

- 플랜은 프로필별 `current/growth_plan.json`에 저장되는 사용자 목표다.
- 빈 목표값은 “현재값 유지”로 동작해야 한다.
- 비용 계산은 항상 `현재 → 목표` 차이다.
- 별, 장비, 무기, 스킬, 능력개방 비용 공식을 UI 레이어에 다시 구현하면 안 된다.
- 단일 학생은 기존 단일 학생 비용 계산, 전체는 기존 합산 계산을 재사용해야 한다.
- 플랜 학생의 수동 순서는 인벤토리의 우선 목표 표시에 영향을 주므로 보존해야 한다.
- 현재처럼 목표 변경이 즉시 저장되는 동작을 바꾸려면 별도의 제품 결정과 마이그레이션/실패 처리 정의가 필요하다.

### 16.5 필요 재화와 인벤토리

- 총 필요량, 보유량, 부족량은 별도 개념으로 유지해야 한다.
- 기존 비용 합계는 총 필요량이며, 인벤토리를 자동 차감한 값으로 의미를 바꾸면 안 된다.
- 계획에 없는 학생의 계산 포함 여부와 레벨·장비·스킬 범위 선택을 보존해야 한다.
- 인벤토리는 프로필별 현재 스냅샷이며 학생·플랜 데이터와 섞어 저장하면 안 된다.
- 계획 부족과 전체 육성 부족의 범위를 구분해야 한다.
- 스캔 데이터가 없을 때 0개 보유와 “알 수 없음”을 동일하게 처리하면 안 된다.

### 16.6 전술대항전과 설정

- 전술 기록, 족보, 시즌, 줄임말, 학생 덱의 의미와 저장 경로를 유지해야 한다.
- 공격 기록, 방어 기록, 족보 모드의 덱 방향을 바꾸면 안 된다.
- 캡처·붙여넣기·폴더·Excel 입력과 기존 기록 편집/삭제/복사 흐름을 보존해야 한다.
- 프로필 생성·전환·삭제는 다른 프로필의 상태를 침범하지 않아야 한다.
- 현재 프로필 데이터 삭제는 명시적 확인이 필요한 파괴적 작업으로 남아야 한다.

## 17. ImageGen 및 목업을 위한 현재 구조 요약

### 17.1 제품 시각 키워드

- Blue Archive 팬 도구
- 어두운 남색 작업공간
- 분홍 포인트와 밝은 하늘색 보조색
- 학생 초상이 중심인 카드 그리드
- 평행사변형/사선 모티프
- 데이터 밀도가 높은 데스크톱 플래너
- 현재/목표/부족을 비교하는 성장 계획 도구

### 17.2 유지할 인지 모델

- 스캔하여 현재 상태를 만든다.
- 학생을 찾아 현재값을 확인한다.
- 학생별 목표를 정한다.
- 여러 학생의 필요량을 합산한다.
- 보유량과 비교해 부족과 우선순위를 판단한다.

### 17.3 목업에서 먼저 풀어야 할 핵심 과제

1. 7개 최상위 탭을 사용자 목표 중심으로 재그룹화할 수 있는가?
2. 프로필·게임 창·스캔 준비 상태를 하나의 명확한 상태 영역으로 만들 수 있는가?
3. 현재값, 목표값, 차이를 한 학생 맥락에서 더 직접적으로 비교할 수 있는가?
4. 필요·보유·부족 중 사용자가 먼저 봐야 할 값을 분명히 할 수 있는가?
5. 학생/필요 재화의 중복 검색·필터 경험을 하나의 공통 패턴으로 만들 수 있는가?
6. 인벤토리의 탐색과 진단을 단계적으로 보여줄 수 있는가?
7. 전술대항전의 입력·기록·분석을 명확한 모드로 분리할 수 있는가?
8. 1366×768에서도 읽을 수 있는 최소 글자와 터치/클릭 목표 크기를 유지할 수 있는가?

## 18. 대표 스크린샷 권장 목록

현재 코드베이스에는 제품 전체를 일관된 조건으로 기록한 최신 스크린샷 세트가 없다. `error_report/`의 이미지는 과거 특정 문제 상황과 게임 위 보조 대화상자를 보여주지만, 현재 기본 탭 노출과 레이아웃을 그대로 대표한다고 보기는 어렵다. 따라서 ImageGen과 목업 전에 아래 조건으로 새 기준 스크린샷을 수집하는 것이 좋다.

### 18.1 최소 대표 세트

1. **스캔 준비 화면**: 프로필과 BA 창이 연결된 상태, 아직 스캔하지 않은 화면
2. **학생 스캔 진행 화면**: 진행률, 현재 학생 카드, 로그, 미리보기가 동시에 보이는 상태
3. **학생 화면**: 보유/미보유 카드가 섞인 그리드와 선택 학생 상세가 보이는 상태
4. **필터 대화상자**: 여러 필터 그룹과 선택 상태가 보이는 화면
5. **플랜 화면**: 3명 이상이 계획에 있고 한 학생의 목표와 필요 재화가 채워진 상태
6. **필요 재화 화면**: 계획 학생과 미계획 학생이 범위에 섞이고 부족 재화가 보이는 상태
7. **인벤토리 화면**: 오파츠 하나를 선택해 좌측 진단, 중앙 목록, 우측 상세가 모두 채워진 상태
8. **전술대항전 화면**: 입력 폼, 기록 목록, 상대 또는 족보 결과가 모두 채워진 상태
9. **설정 화면**: 프로필과 선택된 게임 창 상태가 보이는 화면
10. **플로팅 오버레이 세 상태**: 창 선택 필요, 로비 감시, 스캔 진행
11. **빠른 학생 스캔 대화상자**: 기준 목록과 확인 체크가 모두 보이는 상태
12. **스캔 검토 대화상자**: 정상 행과 경고/수정 행이 함께 있는 상태

### 18.2 촬영 규칙

- 같은 프로필과 같은 예시 데이터 세트를 사용한다.
- 개인 식별 가능 정보와 실제 계정 정보는 가린다.
- 1920×1080 100%를 기준본으로 촬영하고, 1536×864 또는 125% 배율 비교본을 추가한다.
- 각 화면은 빈 상태와 데이터가 채워진 상태를 구분한다.
- 선택, hover, 비활성, 오류, 부족, 미보유 상태가 최소 한 번씩 포함되게 한다.
- Qt 플래너만 찍은 화면과 실제 게임 위 Tk 오버레이가 함께 보이는 화면을 구분한다.
- 파일명에 화면, 상태, 해상도, 배율을 넣는다. 예: `students-selected-1920x1080-100.png`.

### 18.3 ImageGen 입력으로 특히 유용한 4장

모든 화면을 한 번에 제공하기 어렵다면 다음 네 장이 현재 제품의 구조와 정체성을 가장 잘 대표한다.

1. 학생 카드 그리드 + 우측 학생 상세
2. 플랜 학생 목록 + 목표 편집 + 필요 재화 미리보기
3. 인벤토리 진단 + 재화 목록 + 재화 상세
4. 실제 게임 위 스캔 진행 오버레이

이 네 장만으로도 학생 중심 시각 언어, 데이터 밀도, 마스터-디테일 구조, 게임과의 연결 방식을 파악할 수 있다.

## 19. 감사 결론

현재 UI의 가장 큰 강점은 스캔된 현재 상태에서 성장 목표, 필요 재화, 보유량, 부족량까지 실제 기능이 연결되어 있다는 점이다. 학생 초상 카드와 어두운 남색·분홍 시각 언어도 제품 정체성을 분명히 만든다.

가장 큰 문제는 기능 부족이 아니라 정보와 행동의 동시 노출이다. 동일한 시각 위계의 탭과 버튼이 많고, 헤더·패널·밴드가 중첩되며, 현재/목표/필요/보유/부족의 서로 다른 의미가 여러 탭에 흩어져 있다. 1920×1080에서는 많은 정보를 한 화면에 담을 수 있지만, 작은 화면에서는 전체 축소에 의존하여 가독성과 조작성이 급격히 떨어질 가능성이 높다.

따라서 다음 단계의 콘셉트 작업은 기능을 줄이는 방향보다, 핵심 흐름을 `연결 및 스캔 → 학생 확인 → 목표 설정 → 부족 판단`으로 재조직하고 각 화면에서 한 가지 주 작업이 먼저 보이도록 만드는 방향이 적합하다. 시각적으로는 현재의 학생 카드와 게임 친화적 색감을 유지하되, 본문용 타이포그래피, 행동 등급, 상태 색, 패널 깊이, 반응형 레이아웃을 새로 정의해야 한다.
